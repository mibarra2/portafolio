01Introducción

¿Qué es Dine or Dash?

Dine or Dash es un motor de recomendación de restaurantes construido sobre el dataset abierto de Yelp. El proyecto transforma millones de reseñas en sugerencias personalizadas y socialmente relevantes, yendo mucho más allá del clásico ranking de "5 estrellas".

El sistema implementa y compara cinco modelos de recomendación —desde un baseline no personalizado hasta SVD enriquecido con análisis de sentimiento y grafos sociales— evaluando cuál entrega las recomendaciones más útiles y precisas para cada usuario.

El trabajo se focalizó en Philadelphia, la ciudad con mayor densidad de restaurantes y usuarios activos del dataset, filtrando únicamente usuarios que habían dejado al menos 5 reseñas para garantizar suficiente señal de comportamiento.

La paradoja de la elección

"Demasiadas opciones no liberan al usuario — lo paralizan." El surgimiento de plataformas digitales generó una sobrecarga de información donde las reseñas de extraños compiten con la influencia de nuestro círculo social. Dine or Dash modela ambas fuentes para producir recomendaciones más humanas y confiables.

02Motivación

Por qué las 5 estrellas no alcanzan

Los sistemas de rating tradicionales tienen un problema estructural: el sesgo positivo. El promedio de ratings en el dataset es 3.93 sobre 5, y la distribución está fuertemente sesgada hacia calificaciones altas. Los usuarios escriben reseñas principalmente luego de experiencias excepcionales, no promedio.

Además, cada usuario tiene una escala diferente: el usuario 52500 tiene un promedio de 4.60, mientras el usuario 51007 promedia 3.59. Una estrella de uno no equivale a una estrella del otro. El sistema de recomendación debe capturar estas diferencias individuales.

A esto se suma el problema de la densidad de reseñas: con 8,714 usuarios y 3,372 restaurantes en Philadelphia, solo el 0.38% de las combinaciones posibles tiene una reseña real. La escasez de datos obliga a modelos sofisticados que puedan inferir preferencias a partir de señales indirectas.

03Resumen

El proyecto en síntesis

El proyecto diseña, implementa y compara cinco modelos de recomendación sobre el subconjunto de Philadelphia del dataset de Yelp: 111,831 reseñas, 8,714 usuarios únicos y 3,372 restaurantes. Se evalúan mediante RMSE, Precisión, Recall y F1-Score, con el foco principal en el Recall, ya que el objetivo es el descubrimiento —identificar la mayor proporción de restaurantes que el usuario disfrutaría— y no la predicción exacta de la calificación.

El modelo ganador es SVD + Análisis de Sentimiento, que alcanza un Recall de 84.6%. La estrategia recomendada es un sistema híbrido con secciones diferenciadas, donde cada modelo alimenta la sección que mejor se adapta a su naturaleza.

04Metodología

Los cinco modelos implementados

Cada modelo representa una evolución en sofisticación, incorporando progresivamente más fuentes de información: popularidad agregada → comportamiento del usuario → factorización de matrices → red social → texto de reseñas.

1
Rank-Based (Baseline)
Recomendaciones no personalizadas basadas en popularidad y rating promedio. Útil para el problema de cold start: usuarios nuevos sin historial previo.
2
CF User-User (KNN)
Identifica usuarios con patrones de rating similares. Excelente para descubrimiento de experiencias fuera de las categorías habituales del usuario. k=40, similitud MSD.
3
CF Item-Item (KNN)
Encuentra restaurantes similares a los ya visitados y valorados. Útil para sugerencias de "más de lo mismo". k=40, similitud MSD.
4
SVD + Social Graphs
Factorización de matrices SVD con regularización social. Los ratings de los amigos del usuario guían las predicciones. Grafo de 8,714 nodos y 32,917 conexiones.
5
SVD + Sentiment (NLP)
El análisis de sentimiento VADER sobre el texto de la reseña enriquece el rating numérico con la emoción expresada, generando un sentiment_rating compuesto como señal de entrenamiento.
Modelo 5 — Construcción del rating enriquecido con sentimiento
# Paso 1: calcular sentiment score con VADER from nltk.sentiment import SentimentIntensityAnalyzer sia = SentimentIntensityAnalyzer() df['sentiment_score'] = df['review_text'].apply( lambda x: sia.polarity_scores(x)['compound'] * 0.5 + 0.5 # normalizar a [0, 1] ) # Paso 2: construir rating compuesto # sentiment_rating = sentiment_score * 5 → escala 0-5 # rating_final = 0.8 * rating_user + 0.2 * sentiment_rating df['sentiment_rating'] = df['sentiment_score'] * 5 df['rating_final'] = (df['rating_user'] * 0.8 + df['sentiment_rating'] * 0.2) # Paso 3: entrenar SVD sobre el rating enriquecido from surprise import SVD, Dataset, Reader reader = Reader(rating_scale=(1, 5)) data = Dataset.load_from_df( df[['user_id', 'business_id', 'rating_final']], reader ) model = SVD(n_epochs=20, lr_all=0.007, reg_all=0.02) trainset = data.build_full_trainset() model.fit(trainset) # Resultado: Recall = 0.846 | Precisión = 0.728 | RMSE = 1.110

💬
El texto vale más que las estrellas
El modelo con análisis de sentimiento alcanza el mayor Recall (84.6%), superando a todos los modelos puramente numéricos. Las reseñas textuales capturan matices emocionales que el rating de 1–5 no puede expresar, como la diferencia entre un 4 entusiasta y un 4 decepcionado.
👥
El grafo social no cambia significativamente el resultado
SVD con amigos vs SVD simple: diferencia de solo 0.003 en Recall. Los amigos aportan valor cualitativo (las recomendaciones son más confiadas), pero el impacto cuantitativo es marginal, posiblemente por la cantidad de usuarios sin conexiones sociales.
🏆
El modelo User-User es el más robusto en balance
Con F1 de 0.777 y mayor precisión general, el modelo colaborativo usuario-usuario es el más equilibrado. Descubre restaurantes fuera del historial habitual del usuario al identificar perfiles similares, evitando la burbuja de "más de lo mismo".
📊
Asian y Café lideran calidad y volumen
Las cocinas Asiática y Café & Postres mantienen el mejor equilibrio entre volumen de reseñas y rating promedio (>4.07). En cambio, Pizza & Wings y Comida Americana concentran alto volumen pero ratings más bajos (~3.88–3.94), indicando mayor variabilidad en la experiencia.

06Recomendaciones

Estrategia híbrida y próximos pasos

La recomendación principal es implementar un sistema híbrido con secciones diferenciadas donde cada modelo alimenta la sección más alineada con su naturaleza, en lugar de elegir un único modelo ganador.

Arquitectura de secciones recomendada
Cold Start
Tendencias en tu zona
Rank-Based. Para usuarios nuevos sin historial, mostrar los mejores restaurantes por popularidad y rating mínimo garantizado.
Usuarios activos
Basado en tus reseñas
SVD + Sentimiento. Personalización profunda que incorpora la emoción expresada en las reseñas para capturar lo que te gustó de verdad.
Red social
Lo que tus amigos recomiendan
SVD + Friends. Recomendaciones socialmente validadas, especialmente útiles para el componente de confianza interpersonal.
🔬
Incorporar LightFM con metadata del restaurante
Integrar variables como rango de precio, tipo de cocina, capacidad para grupos y ambiente en un modelo híbrido content-based + collaborative para superar las limitaciones de cold start en restaurantes nuevos.
🧠
Evolucionar a Neural Collaborative Filtering (NCF)
Las relaciones no lineales entre usuarios y restaurantes no pueden ser capturadas por SVD. NCF puede aprender patrones de preferencia más complejos, especialmente para usuarios con comportamientos atípicos.
📱
Despliegue local para el mercado uruguayo
Desarrollar una aplicación móvil adaptada al mercado uruguayo que integre la arquitectura híbrida, aprovechando el componente de grafo social que es especialmente relevante en mercados pequeños y de alta confianza interpersonal.
⏱️
Evaluar modelos por etapa del ciclo de vida del usuario
El comportamiento del usuario cambia: los nuevos usuarios son más entusiastas (rating promedio más alto) mientras los usuarios veteranos son más críticos. El sistema debería ajustar qué modelo usar según la antigüedad y actividad del usuario en la plataforma.

Modelo 4 — Construcción del grafo social y regularización
# Construir el grafo de amigos con NetworkX import networkx as nx G = nx.Graph() for _, row in df.iterrows(): user = row['user_id'] friends = str(row['id_friends']).split(',') for friend in friends: friend = friend.strip() if friend: G.add_edge(user, friend) print(f"Grafo: {G.number_of_nodes()} nodos, {G.number_of_edges()} conexiones") # → Grafo: 8714 nodos y 32917 conexiones # Aplicar bono social proporcional (LOG) sobre el SVD # Si el usuario y sus amigos comparten preferencias, # el score SVD se ajusta proporcionalmente a la # intensidad de la conexión social def social_bonus(user_id, business_id, svd_score, G, df): friends_ratings = [] if user_id in G: neighbors = list(G.neighbors(user_id)) friends_ratings = df[ (df['user_id'].isin(neighbors)) & (df['business_id'] == business_id) ]['rating_user'].tolist() if friends_ratings: avg_friends = np.mean(friends_ratings) bonus = np.log1p(len(friends_ratings)) * 0.05 return svd_score + bonus * (avg_friends - svd_score) return svd_score
Limitaciones del proyecto

Item-Item CF: Solo puede sugerir restaurantes similares a los ya visitados. No puede descubrir categorías completamente nuevas para el usuario.

SVD + Friends: Limitado por la densidad de la red social. El 25% de los usuarios no tiene amigos en la plataforma, haciendo que el bono social sea nulo para ese segmento.

Escala geográfica: El modelo fue entrenado y evaluado solo en Philadelphia. La generalización a otras ciudades requeriría reentrenamiento o un enfoque de transfer learning.
