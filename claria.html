01 Introducción

¿Qué es ClarIA?

ClarIA (Claim Assistance Resolution IA) es un agente inteligente diseñado para transformar grandes volúmenes de quejas financieras no estructuradas en insights accionables para la toma de decisiones.

El sistema combina técnicas de Deep Learning —embeddings semánticos, indexación vectorial con FAISS y modelos de lenguaje generativos (LLM)— bajo una arquitectura RAG (Retrieval-Augmented Generation) que permite consultar la base de reclamos en lenguaje natural.

ClarIA fue desarrollado sobre el Consumer Complaint Database de la CFPB (Consumer Financial Protection Bureau), filtrando y procesando específicamente los reclamos dirigidos a entidades bancarias.

¿Qué puede hacer ClarIA?

Responder preguntas como: "¿Cuáles son los principales motivos de reclamo en hipotecas?" o "¿Hay evidencia de fraude digital creciente?" — analizando miles de quejas reales en segundos, sin búsqueda por palabras clave.

02 Motivación

El problema que ClarIA resuelve

Las instituciones financieras reciben miles de quejas de consumidores diariamente, pero carecen de herramientas para identificar automáticamente los problemas estructurales que las generan.

El análisis actual es manual o superficial: se clasifica la queja por producto o categoría, pero no se detectan las causas raíz que podrían corregirse para prevenir futuras quejas. Esto genera un ciclo reactivo costoso: se responde al síntoma, no al problema.

Esta ineficiencia deriva en desperdicio de recursos humanos, deterioro de la experiencia del cliente y riesgos reputacionales acumulados que solo se detectan cuando ya son un problema sistémico.

03 Resumen

El proyecto en síntesis

ClarIA procesa el Consumer Complaint Database de la CFPB, filtrando y depurando un conjunto de 1,941 quejas de entidades bancarias con narrativa textual completa. Sobre este dataset balanceado, se construyó una arquitectura RAG que convierte las quejas en embeddings semánticos mediante el modelo all-MiniLM-L6-v2, indexados en FAISS para búsqueda de alta velocidad.

El agente puede recibir preguntas en lenguaje natural, recuperar los 5 fragmentos de texto más relevantes por similitud semántica y generar una respuesta analítica mediante Llama 3.2. Se evaluaron y compararon cuatro modelos LLM (Llama 3.2-1B, Phi-3 Mini, Mistral y Solar) sobre el mismo pipeline RAG, determinando que Llama 3.2-1B es el modelo con mejor balance entre precisión y ausencia de alucinaciones.

04 Metodología

Pipeline de desarrollo

El desarrollo de ClarIA siguió un pipeline estructurado en seis etapas, desde la depuración del dataset hasta el ajuste fino del modelo.

1
Exploración y limpieza
Auditoría del dataset: 1.28M registros, 30% con narrativa. Se eliminaron nulos, duplicados, outliers de longitud y secuencias de anonimización (XXXX). Filtro: 100–1500 caracteres por comentario.
2
Segmentación y balanceo
Clasificación de empresas por tipo (banco, fintech, prestadora). Consolidación de categorías de producto. Undersampling controlado: máximo 250 quejas por producto, obteniendo 1,941 registros balanceados.
3
Validación con LLM
Uso de Llama 3.2 para clasificar cada queja como ACTIONABLE o NOISE. Solo el 2.47% fue descartado (48 de 1,941), confirmando la alta calidad del corpus filtrado.
4
Chunking optimizado
Experimento comparativo de segmentación: 200, 300 y 400 caracteres con overlap de 20. Se eligieron 300 caracteres por mejor equilibrio entre eficiencia computacional y riqueza contextual. 5,263 fragmentos totales.
5
Indexación semántica
Embeddings con all-MiniLM-L6-v2 (SentenceTransformers). Indexación vectorial con FAISS (IndexFlatL2). Persistencia en disco para evitar reprocesamiento.
6
Ajuste de hiperparámetros
Búsqueda del K óptimo para recuperación: se evaluaron k = 1, 3, 5, 10, 20. El k=5 maximiza el score semántico (0.869) manteniendo precisión. Threshold de seguridad en 1.00 para evitar respuestas sin contexto.
Pipeline RAG — Generación de respuesta con Llama 3.2
class RAGPipeline: def __init__(self, index, metadata_df, model, threshold=1.00): self.index = index self.metadata_df = metadata_df self.model = model self.threshold = threshold # filtro anti-alucinación def generate_answer(self, query): # Paso 1: recuperar los 5 fragmentos más relevantes results = search_complaints(query, self.index, self.metadata_df, self.model, k=5) best_score = results['score'].iloc[0] # Paso 2: bloquear si no hay coincidencia semántica if best_score > self.threshold: return "No encontré quejas relacionadas en mi base de datos.", [] # Paso 3: construir el contexto para el LLM chunks = results['chunk_text'].tolist() context = "\n".join([f"- {c}" for c in chunks]) # Paso 4: prompt estructurado para análisis neutro prompt = f""" SYSTEM: You are a neutral Compliance Senior Assistant. Summarize the main issues from the complaints. Report facts only. CONTEXT: {context} QUESTION: {query} """ response = ollama.chat( model = 'llama3.2:1b', messages= [{'role': 'user', 'content': prompt}], options = {'temperature': 0.2, 'top_p': 0.9} ) return response['message']['content'], chunks

🏦
Productos más reclamados
Hipotecas (Mortgages) y tarjetas de crédito concentran el mayor volumen de reclamos bancarios. Los motivos principales son "Managing an account" y "Account opening", revelando que las causas raíz son problemas de experiencia de usuario y burocracia, no solo fallas financieras.
📈
Pico temporal en 2018
El análisis exploratorio reveló un pico significativo de reclamos hacia 2018, indicativo de un evento sistémico. ClarIA, aplicado en tiempo real, podría detectar estos picos días o semanas antes de que escalen a crisis reputacional.
🎯
Búsqueda por concepto, no por palabras
Al consultar por "interest rates", el sistema identificó quejas sobre tasas en Student Loans y Personal Loans, aunque el usuario no especificó el producto. Los scores entre 0.80–0.83 confirman relación semántica fuerte incluso con vocabulario diferente.
🛡️
Sin alucinaciones en respuestas
El modelo no inventó números, nombres de bancos ni leyes fuera del contexto provisto. El threshold de 1.00 bloqueó eficazmente preguntas fuera de dominio (receta de torta, preguntas en español), devolviendo mensaje de no contexto en lugar de inventar respuestas.

06 Recomendaciones

Próximos pasos y mejoras sugeridas

📊
Dashboard interactivo para analistas
Desarrollar una interfaz visual que permita a analistas de cumplimiento consultar ClarIA en lenguaje natural y explorar los tópicos emergentes con gráficas interactivas, sin necesidad de conocimiento técnico.
⚡
Pipeline de actualización en tiempo real
Implementar un flujo continuo que incorpore nuevas quejas al índice FAISS automáticamente, permitiendo detección proactiva de picos y eventos sistémicos con días de anticipación.
🌐
Ampliar el alcance más allá de bancos
El pipeline actual filtra solo bancos por consistencia analítica. Extender el agente para cubrir fintechs, prestadoras de crédito y otros tipos de instituciones financieras del dataset original.
🔗
Incorporar datos estructurados al contexto RAG
Enriquecer el agente con metadata estructurada (fecha del reclamo, empresa, producto, estado, respuesta de la compañía) para que las respuestas puedan filtrar por variables específicas y no solo por semántica textual.
🔬
Escalar con modelos más grandes o fine-tuning
Evaluar el desempeño con Llama 3.2-8B o modelos especializados en lenguaje financiero. Considerar fine-tuning sobre el corpus de quejas para mejorar la comprensión de terminología bancaria específica.

# Pipeline de indexación vectorial CHUNK_SIZE = 300 CHUNK_OVERLAP = 20 MODEL_NAME = 'all-MiniLM-L6-v2' def run_indexing_pipeline(df): # 1. CHUNKING con metadata splitter = RecursiveCharacterTextSplitter( chunk_size=CHUNK_SIZE, chunk_overlap=CHUNK_OVERLAP ) chunked_data = [] for _, row in df.iterrows(): chunks = splitter.split_text(row['clean_narrative']) for chunk in chunks: chunked_data.append({ "product" : row['Product'], "chunk_text" : chunk }) df_chunks = pd.DataFrame(chunked_data) # 2. EMBEDDINGS semánticos model = SentenceTransformer(MODEL_NAME) embeddings = model.encode(df_chunks['chunk_text'].tolist(), show_progress_bar=True) # 3. INDEXACIÓN con FAISS dimension = embeddings.shape[1] index = faiss.IndexFlatL2(dimension) index.add(np.array(embeddings).astype('float32')) # 4. PERSISTENCIA en disco faiss.write_index(index, "complaints_index.faiss") df_chunks.to_csv("metadata_chunks.csv", index=False) print(f"✅ {len(df_chunks)} fragmentos indexados.") return index, df_chunks
Limitaciones identificadas

Volumen muestral: La necesidad de balancear el dataset redujo 1.28M de registros a 1,941. Escalar el sistema requeriría mayor capacidad computacional para indexar el corpus completo.

Ruido de anonimización: Las secuencias "XXXX" que reemplazan datos confidenciales introducen ruido en los embeddings. La limpieza implementada mitiga el problema pero no lo elimina completamente.

Idioma: El sistema opera en inglés (idioma del dataset). El threshold de 1.00 bloquea correctamente preguntas en español, pero podría extenderse con modelos multilingüe.
